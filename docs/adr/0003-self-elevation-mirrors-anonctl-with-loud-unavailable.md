# Self-elevation mirrors anonctl's sudo re-exec, but fails LOUD when elevation is unavailable

anonseed writes host state under `/etc/anonctl` (root-owned) for the anonctl target, so it needs privilege. The family stance (anonctl) is SELF-ELEVATION, not command-printing: a non-privileged invocation that reaches a root-requiring step re-execs the SAME command through `sudo` (prompting inline in the terminal) and DOES the privileged work itself. anonseed adopts that mechanism VERBATIM (`internal/elevate`, mirroring `../anonctl/elevate.go`): `sudo` (never `pkexec`, which pops a polkit GUI the family avoids); re-exec THIS binary via `os.Executable` (`/proc/self/exe`) preserving argv exactly; a loop-guard sentinel env (`ANONSEED_ELEVATED`) set on the child so a misconfigured sudo can never recurse; and the impure steps (geteuid / sudo lookup / self path / re-exec) behind package-var seams so the decision + argv construction are unit-testable WITHOUT a real sudo, root, or password prompt. We pin the alignment so the two tools behave identically and the hardening cannot drift.

## Considered Options

- **Print "run this as root" hints (the anti-goal).** Rejected: the SPEC (story 10) explicitly wants root-requiring operations to self-elevate rather than print paste-these-commands, mirroring anonctl. Command-printing is exactly the stance this task replaces.
- **`pkexec` / a polkit GUI prompt.** Rejected for the same reason anonctl rejects it: on a tty `sudo` prompts in-terminal, whereas `pkexec` pops a GUI dialog the family avoids.
- **Extract elevation into anoncore and import it.** Rejected for now: anoncore v0.1.0 ships NO elevation package (its `sudoprobe` is a pure classifier of `sudo -l` output, a different concern, not a re-exec mechanism), and anonctl keeps its `elevate.go` private in `package main`. There is nothing to import. anonseed keeps a small aligned copy in `internal/elevate`; if a third consumer appears, extracting into anoncore becomes the right call (same trajectory as the `--allow` guardrail, ADR-0002).
- **Adopt anonctl's mechanism VERBATIM but adjust the three points below (chosen).** The mechanism is identical; only the DECISION SHAPE, the UNAVAILABLE handling, and the (absent) session branch differ, each for a concrete anonseed reason.

## Deliberate divergences from anonctl (the acceptance-required record)

1. **Decision is keyed on REACHING A ROOT-REQUIRING STEP, not on a dispatch-time verb table.** anonctl keys elevation on a fixed `rootRequiringVerbs` map at argv dispatch (read verbs like `list`/`status` never elevate). anonseed's need-for-root is TARGET-dependent (writing the anonctl substrate needs root; a future plan-only / dry-run does not), so `elevate.Ensure` is called at the point the code is about to do the privileged work and returns a `Decision{AlreadyPrivileged | Reexeced+ExitCode | Err}`. Today the gate sits at CLI seed-dispatch (`internal/cli`, `ensureElevated`): every built-in seed writes under `/etc/anonctl`, so a recognised seed IS a root-requiring step. When the `--target` axis (a separate task) and a plan-only path land, the gate can move to fire only for the anonctl target without changing `elevate`.

2. **Elevation-unavailable is a LOUD, HARD FAILURE, not a fall-through.** anonctl, when `sudo` is absent (or self-path unresolved), FALLS THROUGH to run the verb directly and lets the verb surface its own "must be root" error. anonseed has no such downstream verb error at this seam (the write path is a separate task), and the task requires elevation-unavailable to be "a loud, clear failure (never a silent skip or a partial write)". So `Ensure` returns a non-nil `Err` (wrapping `ErrSudoNotFound` for the sudo case) that the caller surfaces and exits non-zero on, BEFORE any `/etc/anonctl` write. This is the load-bearing divergence: on this substrate a silent unprivileged proceed could half-write host state.

3. **No anonctl-session refusal branch.** anonctl refuses (rather than re-execs) when inside an `anonctl use` session, because that dropped anon shell is a sudo dead-end. anonseed has no equivalent in-session concept, so that branch is simply absent.

## Consequences

- **The loop guard preserves safety across the moved gate.** The re-exec re-runs the identical argv, so the elevated child re-enters at the same gate with `ANONSEED_ELEVATED` set; the sentinel check makes it proceed (no second elevation) to the handler as root. Combined with the `euid==0` gate this makes a re-exec loop impossible even under a sudo that failed to elevate. When the sentinel is set but the process is STILL not root, `Ensure` reports `AlreadyPrivileged` (proceed): the downstream privileged write then surfaces its own OS permission error rather than recursing.
- **`ANONSEED_ELEVATED` is a new env name, coherent with anonctl's `ANONCTL_ELEVATED`.** It is the anonseed-namespaced sibling of anonctl's loop-guard sentinel, same role, different tool prefix. It does not re-mean or overlap any existing anonseed concept (CONTEXT.md has no prior sentinel).
- **Testable without sudo/root.** `internal/elevate`'s tests cover the three required cases through the seams (needs-elevation-and-re-execs with argv preserved + sentinel set + exit-code propagated; already-root no-op; elevation-unavailable loud failure), plus the loop guard and the unresolved-self-path failure. `internal/cli`'s tests cover the same wiring at the dispatch seam (argv forwarded, re-exec returns the child code without re-running the handler, unavailable fails loud without running the handler).
- **The gate placement is revisitable, cheaply.** Because the decision is a seam returning a `Decision` (not baked into dispatch), moving it behind `--target` parsing or adding a plan-only opt-out later touches only the call site, not `elevate`.
