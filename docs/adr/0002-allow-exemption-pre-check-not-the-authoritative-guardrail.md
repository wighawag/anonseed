# The `--allow` pre-check is a fail-fast COPY, not the authoritative guardrail

anonseed validates a direct-egress `--allow` value (`internal/allowguard`) into a validated `Exception` (a private-LAN or loopback `IP|CIDR:port` hole) BEFORE writing it into anonctl's `/etc/anonctl/defaults.json` `"allow"` list. This pre-validation is CONVENIENCE (fail-fast, a clear early message), NOT the security boundary. anonctl's own `internal/lanexempt` is the AUTHORITATIVE guardrail: anonctl RE-VALIDATES every default through it on every `add`/`update` (README: "a default exemption is validated through the same guardrail as the flag"), so a bad value anonseed somehow wrote is still rejected by anonctl at apply time. anonseed's check just fails early instead of letting a bad value sit until the next `add` fails. We pin this layering because getting it wrong (treating anonseed's copy as the boundary) would be a silent security regression.

## Considered Options

- **Shell out to anonctl to validate.** Rejected: there is no anonctl pure-validate CLI verb. `anonctl verify` is a LIVE egress prover needing root + a provisioned account (the WRONG tool), and spec story 26 forbids anonseed re-implementing an egress prover. There is nothing to call.
- **Import anonctl's `internal/lanexempt` directly.** Rejected: it is Go-INTERNAL (un-importable) by design. anonctl keeps its guardrail per-tool (anoncore ADR-0001 pins `lanexempt` as anonctl-per-tool, deliberately NOT extracted into anoncore).
- **No pre-check at all (let anonctl reject at apply time).** Rejected as poor UX: a bad `--allow` value would be written and only surface as a confusing failure on the operator's next `anonctl add`, far from where they typed it. The whole point is failing where the value is entered.
- **A small aligned COPY of the POLICY (chosen).** anonseed keeps `internal/allowguard`, mirroring `lanexempt`'s accept/reject matrix VERBATIM, reusing anoncore's `endpoint` primitives (`endpoint.DefaultHost`, `endpoint.Classify` for the 9050/9150 Tor-SOCKS subset) so only the allow-list POLICY is anonseed-local, not the address parsing.

## Consequences

- **Drift risk, accepted and mitigated.** anonseed's copy and anonctl's `lanexempt` can drift if anonctl changes its policy (a range, a blocked loopback port). Mitigations: (1) `allowguard`'s tests are BYTE-ALIGNED with `lanexempt`'s test vectors (same inputs accepted/rejected), so a reviewer diffing the two matrices sees any divergence; (2) the reused overlap (`endpoint.Classify` for Tor ports, `endpoint.DefaultHost` for the loopback host) cannot drift silently, and a package test asserts the reuse still holds; (3) anonctl remains authoritative, so a drift is at worst a UX bug (a value anonseed accepts but anonctl rejects, or vice versa), never a security hole. The value anonctl actually enforces is `lanexempt`'s, always.
- **Only the POLICY is copied, not the parsing.** The address primitives (loopback host, Tor-SOCKS-port recognition) reuse anoncore `endpoint`. The anonseed-local part is the allow-list policy: RFC1918 + link-local containment (LAN branch), the STRICTER loopback port blocklist ({53, 9050, 9150, 9051, 1080}), the `:53`-on-LAN refusal, and the mandatory-port rule.
- **Follow-up: extract into anoncore IF a third consumer appears.** anonctl and anonseed now both hold this policy (anonctl authoritatively, anonseed as a fail-fast copy). A THIRD consumer would tip the balance toward extracting the guardrail into anoncore as shared code, which would REOPEN anoncore ADR-0001 (currently pinning `lanexempt` as anonctl-per-tool). Recorded as `work/notes/ideas/allow-guardrail-extract-to-anoncore.md` so the trigger is not lost. Until then, the aligned copy is the right call: extracting for only two consumers (one of them the authoritative owner) would couple anoncore to a policy anonctl might still evolve.
- **The validated `Exception` (allowguard) is distinct from the declarative `Exception` (seed).** `seed.Exception` (in `internal/seed`) is the DECLARATIVE plan carrier (a raw `Allow` string + `Reason`, JSON-serialised into a `SeedPlan` per ADR-0001); `allowguard.Exception` is the VALIDATED, guardrail-checked value. They share anonseed's ubiquitous word "Exception" (CONTEXT.md) deliberately, split only by layer: a seed applier parses the raw `seed.Exception.Allow` through `allowguard.Parse` and lands the value on success. This is not a duplicated concept, it is the same concept at two layers (intent vs validated value), the same split anonctl draws between its `"allow"` config string and its validated `lanexempt.Exempt`.
